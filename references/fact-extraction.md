# 事实提取模板

从用户提供的截图、录屏或体验记录中提取数据时使用此模板。关键原则：先整理事实，不要直接分析。

## 基础结构硬数据

尽量提取以下8类信息：

| 信息类型 | 需要识别的内容 |
|----------|----------------|
| 页面结构 | 入口位置、页面数量、主操作区、弹窗、页签、按钮、返回路径 |
| 数量规模 | 同屏元素数量、任务数量、订单数量、卡池数量、奖励档位数量、关卡数量等 |
| 容量限制 | 背包容量、体力上限、次数上限、库存上限、格子数量、队伍人数等 |
| 等级层级 | 等级数、阶段数、星级数、品质数、链条长度、解锁层级 |
| 时间规则 | 活动周期、刷新时间、冷却时间、倒计时、恢复时间 |
| 资源规则 | 货币种类、消耗数量、奖励数量、兑换比例、购买价格 |
| 状态变化 | 未解锁、可领取、已完成、已购买、冷却中、售罄、红点提示等 |
| 操作步骤 | 玩家从进入系统到完成一次核心行为需要几步 |
| 付费信息 | 价格、礼包内容、折扣、限购次数、会员权益、广告入口 |

## 可信度标记

每条信息必须标记：

- **已确认**：截图/录屏中直接可见
- **估算**：基于可见信息推算（如"可见5个位置，推测共有8-10个"）
- **待验证**：无法从当前资料确认
- **需补充**：需要用户提供额外截图/录屏

## 联网搜索方向

当用户资料不足时，主动搜索以下类型的公开资料：

1. 官方公告、商店页、版本更新说明
2. 游戏Wiki
3. 玩家攻略（图文/视频）
4. 社区讨论、玩家评论
5. 玩家整理的数据表（订单表、奖励表、概率表、活动表、物品表）
6. 公开截图（商店、礼包、活动、新手流程）
7. 玩家吐槽和负反馈

搜索围绕这些方向补齐：完整结构、核心流程、关键规则、奖励与消耗、付费/广告入口、版本变化、玩家正负反馈、同类系统常见设计。

## 来源类型定义

| 类型 | 说明 |
|------|------|
| 截图确认 | 当前截图中能直接看到 |
| 录屏确认 | 当前录屏中能直接看到 |
| 官方确认 | 来自官方公告、商店页、版本说明 |
| 玩家攻略 | 来自攻略站、玩家帖子、社区整理 |
| 视频资料 | 来自视频内容、标题、简介或截图 |
| 多源交叉 | 至少两个不同来源相互印证 |
| 资料推断 | 根据已知信息推导，没有直接证据 |
| 待验证 | 资料不足，必须进游戏确认 |
| 版本风险 | 资料可能来自旧版本 |
| 冲突信息 | 不同来源之间说法不一致 |

## 冲突处理

不同来源冲突时：
1. 优先保留当前截图事实
2. 公开资料标记版本风险
3. 单独列出"冲突信息表"，格式：

| 冲突内容 | 来源A说法 | 来源B说法 | 建议采用 | 原因 |
